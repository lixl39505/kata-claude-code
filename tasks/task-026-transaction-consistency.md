# task-026-transaction-consistency.md

## 目标

实现: 关键写操作事务一致性收敛

必须包含:

- 检查并收敛所有涉及多表写入的业务操作
- 确保主写入、审计日志、通知、mention 等副作用写入在同一事务内完成
- 防止部分成功导致数据不一致
- 为关键事务路径补齐失败回滚测试

明确不实现:

- 分布式事务
- 消息队列
- 异步任务系统
- 事件总线
- 外部数据库事务协调

## API

### 事务一致性约束

所有关键写操作必须满足事务一致性要求

任意写操作 endpoint

输入:

- 无新增输入

默认值:

- 无

输入约束:

- API 层不得直接开启事务
- Service 层负责组织业务事务
- DB 层提供统一事务能力
- 事务内不得吞掉业务错误

输出:

- 无新增输出结构

输出约束:

- 事务失败时必须返回统一错误结构
- 不得出现主数据成功但审计 / 通知 / mention 缺失的情况
- 不得出现副作用数据成功但主数据失败的情况

## 事务覆盖范围

必须覆盖:

- 创建 Issue + 审计日志
- 更新 Issue 状态 + 审计日志
- 更新 assignee + 审计日志 + 通知
- 创建评论 + mention 记录 + mention 通知
- 删除评论 + mention 清理 + 审计日志
- 添加 Project 成员 + 审计日志
- 移除 Project 成员 + 审计日志
- 批量更新 Issue + 每条审计记录

## 业务规则

事务规则:

- 主业务写入与副作用写入必须在同一事务
- 任意一步失败，整个操作必须回滚
- 事务边界必须清晰集中，避免嵌套事务造成不可预测行为
- 批量操作必须整体成功或整体失败，不允许静默部分成功
- 审计日志仍然只允许追加，不允许修改或删除

错误处理规则:

- 业务错误应正常回滚并返回对应错误码
- 未预期错误应回滚并返回 INTERNAL
- 不得泄露 SQL 或 stack trace

## 测试用例

### Issue 创建事务回滚

创建 Issue 成功后审计写入失败时，Issue 不得残留

### 状态更新事务回滚

状态更新后审计写入失败时，状态不得被修改

### assignee 通知事务回滚

assignee 更新后通知写入失败时，assignee 不得被修改

### 评论 mention 事务回滚

评论创建后 mention 写入失败时，评论不得残留

### mention 通知事务回滚

mention 记录写入后通知写入失败时，评论与 mention 均不得残留

### 删除评论一致性

删除评论时 mention 清理与审计日志必须保持一致

### Project 成员事务回滚

添加或移除成员时审计写入失败，成员关系不得变化

### 批量操作整体回滚

批量更新中任意一条失败时，所有 Issue 均不得被修改

### 错误结构一致

事务失败后返回统一错误结构
